# Design System

Framework for design system decisions. Defines WHAT every project must decide ‚Äî the actual values live in CLAUDE.md, generated by `/setup-project`.

---

## Design Tokens

Every project defines these token categories before building any UI.

### Color Semantics

Map color variants to meaning. The intent layer ‚Äî what each color communicates in this project:

- **Primary action**: buttons, active states, links, focus rings
- **Success / active**: confirmations, active statuses, positive indicators
- **Warning / pending**: alerts, pending statuses, attention-needed states
- **Danger / error**: destructive actions, error states, validation failures
- **Neutral**: borders, muted text, secondary surfaces, disabled states
- **Accent** (optional): highlights, decorative elements, brand differentiation from primary

Each maps to a single color family. Primary and accent are project-specific choices; danger is always red; the rest depend on project personality.

### Spacing Rhythm

A four-level scale applied consistently across all pages:

| Level | Where it applies |
|-------|-----------------|
| **Page sections** | Between major content blocks (header ‚Üí content, content ‚Üí footer) |
| **Groups** | Between related clusters within a section (form section ‚Üí form section) |
| **Items** | Between individual elements within a group (field ‚Üí field, card ‚Üí card) |
| **Card internals** | Padding inside contained elements (card padding, modal padding) |

Actual values depend on the project's density choice. Pick values once, use everywhere.

### Density

A project-wide choice that cascades into spacing values:

- **Compact**: tight spacing, dense information display ‚Äî suits data-heavy apps, admin panels, dashboards
- **Comfortable**: generous whitespace, breathing room ‚Äî suits consumer-facing apps, marketing, content-heavy pages

Pick one. Mixing densities within a project creates visual inconsistency.

## Foundational Component Inventory

Standard inventory of PROJECT-OWNED Blade components in `resources/views/components/`. Include the ones that match your project's page types ‚Äî not every project needs all of them.

### page-header

Page-level heading with optional actions.

- **Props**: `title` (string), `subtitle` (string, optional)
- **Slots**: `actions` ‚Äî right-aligned buttons/controls
- Appears once at the top of every page

### section-group

Groups related content within a page.

- **Props**: `heading` (string), `description` (string, optional)
- **Slots**: default ‚Äî group content
- Use between logical sections (form groups, settings categories, dashboard regions)

### stat-card

Single metric display for dashboards and summaries.

- **Props**: `label` (string), `value` (string/number), `icon` (string, optional)
- Arrange in rows for KPI sections

### status-badge

Semantic status indicator that maps to project-defined color semantics.

- **Props**: `status` (string) ‚Äî maps to a color semantic (e.g., "active" ‚Üí success, "pending" ‚Üí warning)
- Color mapping is project-specific, defined in CLAUDE.md

### empty-state

Placeholder for sections with no data.

- **Props**: `icon` (string, optional), `title` (string), `description` (string, optional)
- **Slots**: `action` ‚Äî call-to-action button
- Every list, table, and collection needs an empty state

### info-row

Label-value pair for detail pages and read-only displays.

- **Props**: `label` (string)
- **Slots**: `default` ‚Äî value content (can be text, badges, links)
- Stack vertically in detail sections

## Page Archetypes

Structural patterns for common page types. Every page in the project should follow one of these.

### List Page

Header (title + primary action) ‚Üí filters/search bar ‚Üí content (card grid or table rows) ‚Üí pagination.

Content can be cards or table depending on data density. Filters collapse on mobile. Always include empty state.

### Detail Page

Hero section (key info at a glance: title, status, primary metrics) ‚Üí tabbed or sectioned content with info-rows, related lists, activity.

Hero keeps the most important information visible without scrolling. Tabs for pages with many content categories; sections for simpler detail pages.

### Form Page

Header (title + cancel) ‚Üí section groups with fields ‚Üí sticky/fixed footer with submit + secondary actions.

Group related fields in section-groups. Primary action is rightmost in footer. Validate on blur, show errors inline.

### Dashboard

KPI row (stat-cards) ‚Üí primary content (main chart or list) ‚Üí secondary content (recent activity, quick actions).

KPIs are the first thing users see. Limit to 3-5 stats. Primary content gets the most space.

## Designer Boundary

Convention for separating design work from application logic.

### Designers Work In

- `resources/views/components/` ‚Äî Blade components with props/slots, composing framework primitives
- `tailwind.config.js` ‚Äî theme extensions (colors, spacing, fonts)
- Blade views ‚Äî layout, structure, and visual refinement of existing pages

### Designers Do NOT Touch

- Livewire component logic (PHP classes, computed properties, actions)
- `wire:*` directives ‚Äî these are application behavior, not design
- Routes, controllers, middleware ‚Äî backend concerns
- PHP beyond Blade template syntax

### Workflow

1. Developer builds functional screens using vanilla framework components ‚Äî correct data, correct behavior, rough layout
2. Designer refines the component layer ‚Äî extracts patterns, polishes spacing, establishes visual consistency
3. Developer integrates any new components back into Livewire views
